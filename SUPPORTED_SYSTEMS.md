# Supported Systems

## Implemented automatic-migration contract

- Local Windows, macOS, or Linux; Node.js 24.18 or newer within major 24; Git; pnpm 11.9.0; authenticated Codex.
- TypeScript or JavaScript, ESM or CommonJS, npm or pnpm.
- Native `node:crypto` RSA signing and verification with defensible RSA call/key/contract evidence. A hash name such as SHA256 alone is ambiguous, not proof of RSA.
- Reviewed version-1 `quantum-twin.config.json` with commands, writable/protected paths, source boundaries, dependency policy, ML-DSA-65 context, limits, and contained external compatibility harness.
- Direct Cutover and Compatibility Bridge targeting native `ml-dsa-65`.

Local intake modes are the independent public demo, strict public GitHub HTTPS URL, trusted local folder path, Chromium folder import, and ZIP import. Intake and analysis never execute code. Migration requires no blockers and three explicit acknowledgements.

## Discovery only

TLS/X.509, JWT, Cloud KMS, HSM/PKCS#11, third-party cryptography, Java, Python, .NET, Go, and Rust produce evidence and adapter/human requirements, never patches.

## Explicitly unsupported

- Private repositories, credentials, SSH, arbitrary Git hosts/protocols, local-network URLs, redirects, and automatic pull requests.
- Hosted/Vercel repository intake or execution.
- Automatic migration without both native RSA signing and verification evidence, a reviewed contract, and external harness.
- Hostile-repository sandboxing; HSM/KMS/certificate/TLS migration; non-Node automatic migration.
- Formal verification, certification, side-channel proof, guaranteed security, whole-system quantum safety, or deployment approval.
